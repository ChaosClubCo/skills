# Codex Bundles

This guide covers the 5 bundles available for the OpenAI Codex platform. Each bundle is located under `CodexSkills/bundles/` and contains JSON files formatted for the Responses API, GPT Builder, or Agent Builder.

---

## Overview

| Bundle | Skills | Format | Target |
|--------|-------:|--------|--------|
| [responses-api-full](#responses-api-full) | 508 | .response.json | Complete Responses API library |
| [by-category](#by-category) | 508 | .response.json | Browsable full library |
| [enterprise-assistants](#enterprise-assistants) | 53 | .response.json | Enterprise teams |
| [gpt-builder-50](#gpt-builder-50) | 50 | .gpt.json | Custom GPT creators |
| [agent-builder-20](#agent-builder-20) | 20 | .agent.json | Agent Builder developers |

---

## Codex File Formats

Codex bundles use three different JSON formats depending on the target platform feature:

**Responses API (.response.json)** -- Structured JSON for the OpenAI Responses API. Contains system prompts, parameters, and tool configurations. Generated by `build_codex_response_json()` in `populate_all.py`. Used by: responses-api-full, by-category, enterprise-assistants.

**GPT Builder (.gpt.json)** -- Configuration files for OpenAI's GPT Builder interface. Each file includes name, description, instructions, conversation starters, and capability flags. Used by: gpt-builder-50.

**Agent Builder (.agent.json)** -- Agent definition files for OpenAI's Agent Builder. Includes model selection, tool definitions, handoff descriptions, and metadata. Used by: agent-builder-20.

### Deployment

```bash
# Deploy full Responses API bundle to Codex CLI
cp -r CodexSkills/bundles/responses-api-full/ ~/.codex/skills/

# Or deploy the entire CLI skill set (includes config)
cp -r CodexSkills-CLI/.codex ~/.codex
```

### Configuration Tips

The Responses API format includes model and parameter suggestions per skill, but your application can override these at call time. The default model assignments come from `CODEX_MODELS` in `lib/config.py` and vary by category.

For production deployments, consider:

- Pinning a specific model version (e.g., `gpt-4.1` instead of the default)
- Adjusting temperature per use case
- Adding custom tool definitions for skills that reference external services

### When to Choose

Use responses-api-full when building applications on the Responses API that need the full skill catalog. This is the Codex equivalent of Gemini's gems-full bundle. Best for API developers who will programmatically select skills at runtime.

### Configuration Tips

For enterprise deployments on the OpenAI platform:

- Use a dedicated API key with appropriate rate limits and access controls

## gpt-builder-50

**Path:** `CodexSkills/bundles/gpt-builder-50/`
**Skills:** 50
**Format:** GPT Builder JSON

### Contents

The 50 highest-scoring skills across all categories, formatted as GPT Builder configurations. Each file is a JSON object ready to import into the GPT Builder interface.

### JSON Structure

Each `.gpt.json` file contains:

```json
{
  "name": "Api Design Principles",
  "description": "Guidelines for designing consistent, scalable REST APIs...",
  "instructions": "...(full skill body)...",
  "conversation_starters": [
    "Help me with api design principles",
    "Explain how to use api design principles",
    "What are best practices for api design principles?",
    "Review my api design principles setup"
  ],
  "capabilities": {
    "web_browsing": false,
    "code_interpreter": true,
    "image_generation": false,
    "file_upload": true
  },
  "category": "technical"
}
```

Capability flags are set per category:

| Category | Web Browsing | Code Interpreter | Image Generation | File Upload |
|----------|:------------:|:----------------:|:----------------:|:-----------:|
| technical | No | Yes | No | Yes |
| ai-agents | Yes | Yes | No | Yes |
| strategy | Yes | Yes | No | Yes |
| creative | No | Yes | Yes | Yes |
| operations | No | Yes | No | Yes |
| industry | Yes | Yes | No | Yes |

### Deployment

To use these with GPT Builder:

1. Open the GPT Builder at https://chat.openai.com/gpts/editor
2. Open the `.gpt.json` file for the skill you want to create
3. Copy the `name` into the Name field
4. Copy the `description` into the Description field
5. Copy the `instructions` into the Instructions field
6. Add the `conversation_starters` as suggested prompts
7. Configure capabilities as specified in the JSON

Or browse from the command line:

```bash
# List available GPT configs
ls CodexSkills/bundles/gpt-builder-50/

# View a specific GPT configuration before importing
cat CodexSkills/bundles/gpt-builder-50/api-design-principles.gpt.json | python -m json.tool
```

### Configuration Tips

The conversation starters are auto-generated from the skill name using these templates:

```
"Help me with {name}"
"Explain how to use {name}"
"What are best practices for {name}?"
"Review my {name} setup"
```

For production GPTs, consider replacing them with more specific prompts relevant to your use case. For example, instead of "Help me with api design principles," use "Design a REST API for a user management service."

The `capabilities` flags provide sensible defaults, but you can adjust them in GPT Builder based on your GPT's actual needs.

### When to Choose

Use gpt-builder-50 when creating custom GPTs in OpenAI's GPT Store or for internal team use. Each file provides a complete starting point -- paste the instructions, set the capabilities, and publish.

## agent-builder-20

**Path:** `CodexSkills/bundles/agent-builder-20/`
**Skills:** 20
**Format:** Agent Builder JSON

### Contents

The top 20 skills from the `ai-agents` category, formatted as Agent Builder configuration files. Each file defines a complete agent with model selection, tool access, and multi-agent handoff support.

### JSON Structure

Each `.agent.json` file contains:

```json
{
  "name": "Autonomous Agent Design Agent",
  "description": "Patterns for designing self-directed AI agents...",
  "instructions": "...(full skill body)...",
  "model": "gpt-4.1",
  "tools": [
    {"type": "code_interpreter"},
    {"type": "web_search"}
  ],
  "handoff_description": "Patterns for designing self-directed AI agents that can plan, ...",
  "metadata": {
    "category": "ai-agents",
    "version": "1.0.0"
  }
}
```

Field details:

- `model` -- Set per-category using `CODEX_MODELS` from `lib/config.py`
- `tools` -- Set per-category using `CODEX_TOOLS` from `lib/config.py`
- `handoff_description` -- First 200 characters of the skill description, used by multi-agent orchestration systems to decide when to hand off to this agent

### Directory Structure

```
CodexSkills/bundles/agent-builder-20/
  autonomous-agent-design.agent.json
  multi-agent-orchestration.agent.json
  agent-memory-management.agent.json
  ...
  (20 files)
```

### Deployment

```bash
# Browse available agent configs
ls CodexSkills/bundles/agent-builder-20/

# View a specific agent configuration
cat CodexSkills/bundles/agent-builder-20/autonomous-agent-design.agent.json | python -m json.tool
```

To use these files with OpenAI's Agent Builder, load each JSON as an agent definition. The `handoff_description` field enables multi-agent routing -- an orchestrator agent can read these descriptions to decide which specialist agent should handle a given request.

### Configuration Tips

For multi-agent systems, load multiple agent configs and wire them together using the handoff mechanism. Each agent's `handoff_description` tells the orchestrator what this agent is good at.

Adjust the `model` field based on your cost and latency requirements. The default `gpt-4.1` provides the best quality, but `gpt-4.1-mini` may be sufficient for simpler agent tasks.

Consider adding custom tools beyond the defaults. The `tools` array accepts any tool definition supported by the Agent Builder API.

### When to Choose

Use agent-builder-20 when building multi-agent systems with OpenAI's Agent Builder. Each file provides a complete agent definition with tool access, model configuration, and orchestration metadata. This is the Codex counterpart to Gemini's agent-chains-20 and Copilot's coding-agent-20 bundles.

## Regenerating Codex Bundles

All Codex bundles are generated by `populate_all.py` during the pipeline's bundle phase:

```bash
PYTHONIOENCODING=utf-8 python populate_all.py --phase bundles
```

To preview without writing:

```bash
PYTHONIOENCODING=utf-8 python populate_all.py --phase bundles --dry-run
```
