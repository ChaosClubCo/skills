# Project Management

## Overview

Project management is the disciplined application of knowledge, skills, tools, and techniques to project activities to meet requirements. For SMBs, effective project management means delivering results on time, within budget, and to specification while maintaining flexibility and minimizing overhead.

This skill provides frameworks and templates scaled appropriately for small and medium businesses, avoiding enterprise-level complexity while maintaining professional standards. The goal is to establish repeatable processes that grow with your organization.

### Why Project Management Matters for SMBs

- **Resource Optimization**: SMBs have limited resources; effective PM ensures maximum utilization
- **Risk Reduction**: Early identification and mitigation of risks prevents costly failures
- **Stakeholder Alignment**: Clear communication keeps everyone moving in the same direction
- **Competitive Advantage**: Professional project delivery builds client trust and repeat business
- **Scalability**: Documented processes enable growth without chaos

## When to Use

### Primary Triggers
- Starting any new initiative requiring coordination across people or departments
- Client engagements with defined deliverables and timelines
- Internal improvement projects (system implementations, process changes)
- Product launches or service expansions
- Organizational changes requiring structured rollout

### Specific Use Cases
- "We need to plan a new software implementation"
- "How do I track multiple projects simultaneously?"
- "What methodology should we use for this client project?"
- "I need to create a project charter for stakeholder approval"
- "Help me build a work breakdown structure"
- "We need better risk management processes"
- "How do I communicate project status effectively?"

## Core Processes

### 1. Methodology Selection Framework

#### Waterfall Methodology
**Best For**: Projects with well-defined requirements, regulatory compliance needs, fixed budgets

```
Phase Structure:
1. Initiation → 2. Planning → 3. Execution → 4. Monitoring → 5. Closure

Characteristics:
- Sequential phases with formal gate reviews
- Comprehensive upfront documentation
- Change control processes for scope modifications
- Clear milestones and deliverables per phase
```

**Waterfall Decision Criteria**:
- Requirements are stable and well-understood
- Regulatory or compliance requirements demand documentation
- Client expects traditional deliverable-based billing
- Project duration is 3+ months with clear phases
- Technology and approach are proven

#### Agile Methodology
**Best For**: Projects with evolving requirements, innovation focus, collaborative clients

```
Sprint Structure (2-week recommended for SMBs):
Week 1: Sprint Planning → Development → Daily Standups
Week 2: Development → Sprint Review → Retrospective

Key Ceremonies:
- Sprint Planning (2-4 hours): Define sprint goals and backlog
- Daily Standup (15 min): Progress, blockers, plans
- Sprint Review (1 hour): Demo completed work to stakeholders
- Retrospective (1 hour): Team improvement discussion
```

**Agile Decision Criteria**:
- Requirements are likely to change or emerge
- Client wants frequent visibility and input
- Project involves innovation or uncertainty
- Team is co-located or has good remote collaboration
- Rapid delivery of value is prioritized

#### Hybrid Methodology
**Best For**: Most SMB projects - combines structure with flexibility

```
Hybrid Framework:
┌─────────────────────────────────────────────────────────┐
│ Phase 1: Waterfall Initiation & Planning                │
│ - Project Charter (fixed)                               │
│ - High-level requirements (fixed)                       │
│ - Budget and timeline framework (fixed)                 │
└─────────────────────────────────────────────────────────┘
                           ↓
┌─────────────────────────────────────────────────────────┐
│ Phase 2: Agile Execution                                │
│ - Sprint-based delivery                                 │
│ - Iterative refinement                                  │
│ - Flexible scope within boundaries                      │
└─────────────────────────────────────────────────────────┘
                           ↓
┌─────────────────────────────────────────────────────────┐
│ Phase 3: Waterfall Closure                              │
│ - Formal acceptance                                     │
│ - Documentation handoff                                 │
│ - Lessons learned                                       │
└─────────────────────────────────────────────────────────┘
```

### 3. Work Breakdown Structure (WBS) Creation

#### WBS Development Process

```
Step 1: Start with Deliverables
- Identify all major deliverables from the scope statement
- These become Level 1 of your WBS

Step 2: Decompose to Work Packages
- Break each deliverable into components
- Continue until work packages are:
  - 8-80 hours of effort (for SMBs, 4-40 hours is acceptable)
  - Assignable to one person or team
  - Estimable with reasonable accuracy

Step 3: Apply the 100% Rule
- Child elements must represent 100% of parent work
- No gaps, no overlaps

Step 4: Number the WBS
- Use hierarchical numbering (1.0, 1.1, 1.1.1)
- Enables tracking and reporting
```

#### WBS Template Structure

```
1.0 Project Name
    1.1 Project Management
        1.1.1 Project Planning
        1.1.2 Status Reporting
        1.1.3 Risk Management
        1.1.4 Change Management
        1.1.5 Project Closure

    1.2 [Deliverable 1]
        1.2.1 [Component 1.1]
            1.2.1.1 [Work Package]
            1.2.1.2 [Work Package]
        1.2.2 [Component 1.2]

    1.3 [Deliverable 2]
        1.3.1 [Component 2.1]
        1.3.2 [Component 2.2]

    1.4 [Deliverable 3]
        1.4.1 [Component 3.1]
        1.4.2 [Component 3.2]
```

### 4. Resource Allocation Framework

#### Resource Planning Matrix

```markdown
| Resource | Project A | Project B | Project C | Available | Utilization |
|----------|-----------|-----------|-----------|-----------|-------------|
| [Name] | 20 hrs | 10 hrs | 0 hrs | 40 hrs | 75% |
| [Name] | 15 hrs | 15 hrs | 10 hrs | 40 hrs | 100% |
| [Name] | 0 hrs | 20 hrs | 20 hrs | 40 hrs | 100% |
```

#### Resource Loading Guidelines for SMBs

```
Optimal Utilization Targets:
- Individual Contributors: 75-85% (allows for overhead, meetings, unexpected work)
- Managers: 50-60% (accounts for management responsibilities)
- Specialists: 60-70% (allows for context switching between projects)

Warning Thresholds:
- Over 90%: Risk of burnout, no capacity for issues
- Under 60%: May indicate underutilization or project gaps
```

### 5. Risk Management Process

#### Risk Identification Techniques

```
1. Brainstorming Sessions
   - Include diverse team perspectives
   - Use prompts: "What could go wrong with..."
   - No filtering during ideation

2. Checklist Review
   - Technical risks
   - Resource risks
   - Schedule risks
   - Budget risks
   - External/vendor risks
   - Stakeholder risks

3. Lessons Learned Review
   - Review past project issues
   - Interview experienced team members
```

## Metrics & KPIs

### Project Health Metrics

```
Schedule Performance:
- Schedule Variance (SV) = Earned Value - Planned Value
- Schedule Performance Index (SPI) = EV / PV
  - SPI > 1.0 = Ahead of schedule
  - SPI < 1.0 = Behind schedule

Cost Performance:
- Cost Variance (CV) = Earned Value - Actual Cost
- Cost Performance Index (CPI) = EV / AC
  - CPI > 1.0 = Under budget
  - CPI < 1.0 = Over budget

Simplified for SMBs:
- % Complete vs % Timeline Elapsed
- Actual Spend vs Budgeted Spend
- Deliverables Completed vs Planned
```

### Team Performance Metrics

```
- Velocity (Agile): Story points completed per sprint
- Cycle Time: Time from start to completion of work items
- Defect Rate: Issues found per deliverable
- Team Utilization: Billable/productive hours vs available
```

### Stakeholder Satisfaction

```
- Net Promoter Score (NPS) at project close
- Milestone acceptance rate (first-time approvals)
- Change request frequency (scope stability indicator)
- Communication feedback surveys
```

## Common Pitfalls

### 1. Scope Creep
**Problem**: Uncontrolled expansion of project scope
**Prevention**:
- Document scope clearly in charter
- Implement change control process
- Educate stakeholders on impact of changes
- Track and report scope changes

### 2. Poor Estimation
**Problem**: Underestimating time, cost, or complexity
**Prevention**:
- Use historical data from similar projects
- Apply contingency buffers (15-25%)
- Break work into smaller, estimable pieces
- Get multiple estimation inputs

### 3. Inadequate Risk Management
**Problem**: Risks materialize without mitigation
**Prevention**:
- Regular risk review sessions
- Assign risk owners with accountability
- Track early warning indicators
- Maintain contingency budget/time

### 4. Communication Gaps
**Problem**: Stakeholders surprised by issues or changes
**Prevention**:
- Establish communication plan upfront
- Proactive status reporting
- Escalate issues early
- Document decisions and share broadly

### 5. Resource Overallocation
**Problem**: Team members spread too thin
**Prevention**:
- Realistic capacity planning
- Buffer for unexpected work
- Clear prioritization when conflicts arise
- Regular workload reviews

### 6. Insufficient Stakeholder Engagement
**Problem**: Key stakeholders not involved until problems arise
**Prevention**:
- Stakeholder analysis at project start
- Regular touchpoints with key stakeholders
- Clear escalation paths
- Involve stakeholders in reviews/decisions
